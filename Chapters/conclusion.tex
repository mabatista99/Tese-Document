\chapter{Conclusions and Future Work}
\label{cha:Conclusions}

\section{Synthesis}

This thesis presented the design, implementation, and evaluation of a cloud-based \gls{IoT} event management platform for monitoring and surveillance. The platform was built as a proof-of-concept to explore how a microservice architecture with event-driven communication can address the main challenges of \gls{IoT} monitoring: accepting data from heterogeneous devices, processing it with low latency, scaling under load, and continuing to operate when cloud connectivity is lost.

A requirements analysis produced 14 functional requirements and 7 non-functional requirements, organized around five use cases. These guided the architecture, which separates the platform into five layers: external, edge, communication, backend services, and storage. The cloud deployment runs nine NestJS microservices connected through Apache Kafka. A protocol adapter accepts \gls{MQTT}, \gls{HTTP}, and \gls{CoAP} ingestion so that devices using different protocols converge on the same event pipeline. A schema-agnostic ingestion model discovers and normalizes telemetry fields without predefined device schemas. A \gls{CQRS} split keeps write-heavy ingestion separate from user-facing reads.

An edge computing node extends the platform with offline resilience. It runs a standalone application with a local Mosquitto broker and SQLite database, ingesting telemetry and evaluating cached \texttt{CRITICAL} rules while disconnected. RS256 asymmetric \gls{JWT} signing enables offline device authentication. When connectivity returns, the edge synchronizes buffered data to the cloud.

The platform was evaluated through five experiments: a baseline performance study across the three ingestion protocols, a scaling study on the \gls{MQTT} path, an offline resilience test, an extensibility test with an additional Python vision service, and a device heterogeneity test with a Raspberry Pi sensor.

\section{Main Results and Challenges}

The baseline experiment showed that all three ingestion protocols work but differ in throughput capacity. \gls{MQTT} tracked the offered load up to 5\,000\,msg/s and was the strongest protocol. \gls{HTTP} settled around 4.3--4.6k\,msg/s at high load, while \gls{CoAP} saturated first at 2\,500\,msg/s. At moderate load, all three protocols kept p95 ingress latency below 5\,ms. The scaling study addressed the \gls{MQTT} bridge bottleneck by adding replicas and Kafka partitions, increasing throughput by 174\%. The bottleneck then moved to shared host resources, confirming that the architecture supports service replication but a single host limits near-linear scaling.

The three validation experiments confirmed the platform's design goals. The offline resilience test showed that the edge node preserved all buffered telemetry and alerts during a five-minute disconnection, with no observed data loss after reconnection. The extensibility test added a Python-based vision service through Kafka topics without modifying any existing service, confirming that the event backbone is language-agnostic. The heterogeneity test used a Raspberry Pi with an ultrasonic sensor, and the platform discovered the new fields automatically. Across all experiments, the platform handled three different producer types without device-specific code changes.

The main challenges came from scope and evaluation constraints. Building nine microservices, a bridge, an edge node, and a web application as a single developer required trade-offs between depth and breadth. The single-host deployment made it difficult to isolate bottlenecks cleanly, and the load generator running on the same machine introduced an resource overhead that would not exist in a distributed setup. The edge evaluation ran in containers rather than on physical hardware, so resource feasibility on a real edge device was not confirmed. Security was implemented across the platform but not benchmarked separately.

\section{Future Work}
\label{sec:FutureWork}

Several directions for future development were identified during this work.

The platform currently runs as Docker Compose containers on a single host, which limits horizontal scaling. Migrating to \textbf{Kubernetes} would enable horizontal pod autoscaling, service discovery, rolling updates, and resource isolation across nodes. Since each microservice already runs as an independent container, the transition would be straightforward and would remove the single-host bottleneck identified in the scaling study.

A \textbf{reverse proxy layer} with \gls{TLS} termination, using a tool such as Nginx, placed in front of the services would improve both security and scalability. It would provide \gls{HTTPS} for all external-facing endpoints, load balancing across service replicas, and rate limiting. In a Kubernetes deployment, an Ingress controller would fill this role natively.

Advancing to \textbf{machine learning-based anomaly detection}, such as isolation forests, autoencoders, or \gls{LSTM} networks, would complement the current Z-score approach by detecting non-linear and seasonal patterns. The event-driven architecture supports this directly: a new ML consumer could subscribe to the existing Kafka topics without modifying any component.

Integrating \textbf{business intelligence tools} such as Apache Superset or Grafana would add historical trend analysis, custom reporting dashboards, and data exploration over the stored telemetry. Both tools connect natively to PostgreSQL-compatible databases like TimescaleDB, so the integration would require no changes to the storage layer.

A \textbf{stateful alerting system} with enter and exit states per rule would reduce alert noise by triggering only on threshold crossings and introducing hysteresis to prevent rapid toggling.

The platform currently supports a single edge node. A centralized \textbf{edge fleet management} system would enable discovery of multiple edge nodes, centralized configuration distribution, health monitoring, and coordinated updates across distributed deployments.
